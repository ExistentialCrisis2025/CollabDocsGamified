## 📖 About

Task Forge is a full-stack productivity platform that turns everyday task management into a game. Instead of a flat to-do list, users work from a Kanban board, run focus sessions with a Pomodoro timer, and get rewarded with XP, badges, and streaks for staying consistent. A scheduled AI service reviews each user's week and writes a plain-language summary of their progress, so the data isn't just stored — it's actually reflected back to the user.

---

## ✨ Features

### 📋 Task Management
A drag-and-drop **Kanban board** organizes tasks into workflow stages (e.g. To Do → In Progress → Done). Task position is persisted server-side, so reordering survives refreshes and devices. Task creation/editing runs through a dedicated validation schema layer, keeping bad data out of the database before it ever reaches the gamification logic that depends on it.

### 🍅 Focus Tools
A built-in **Pomodoro timer** structures work into focused intervals with breaks, and a **lo-fi music player** runs alongside it. These live next to the Kanban board rather than in a separate app, on the theory that focus tools are most useful when they're one click away from the work itself, not in a separate tab.

### 🏆 Gamification Engine
This is the core of the app, made up of several connected systems:

- **XP & Levels** — every completed task or quest grants XP, logged in a full XP history table (not just a running total), so progress is auditable and a level-up overlay can trigger a celebratory animation at the right threshold instead of a silent number change.
- **Quests** — short-term goals (daily/weekly) are auto-generated by a scheduled job rather than hand-authored, so the game keeps producing new objectives without manual content upkeep.
- **Badges** — awarded automatically by a badge-evaluation service that checks user activity against milestone conditions (streak length, tasks completed, etc.) on a schedule, with an initial badge set seeded at setup so the system has rewards available from day one.
- **Streaks** — daily activity is evaluated by a cron job; a **streak shield** mechanic gives users a one-day grace period so a single missed day doesn't erase weeks of consistency — a deliberate choice to reduce the punishing all-or-nothing feeling of typical streak systems.
- **Leaderboard** — aggregates XP and streak data across users to add light social competition on top of personal progress tracking.

### 🤖 AI Weekly Reports
A scheduled job collects a user's week of activity, builds a structured prompt from it, and generates a natural-language report summarizing trends — what got done, what slipped, and how XP/streaks moved. Keeping prompt construction in its own module (separate from the report generation and scheduling logic) makes it easy to refine the wording or swap models without touching the scheduling or data-gathering code.

### 📊 Analytics Dashboard
A dedicated analytics page visualizes XP growth, task completion rates, and streak history over time, giving users a longer-term view that the day-to-day Kanban/Pomodoro screens don't show.

### 👤 Accounts & Integrations
- JWT-based authentication with route-level protection on the frontend.
- **Google OAuth** for one-click sign-in, in addition to standard email/password.
- **Calendar integration** with stored OAuth tokens, so tasks or quests can sync with external calendar events.
- **Timezone-aware** scheduling, so cron-driven quests, streaks, and reports evaluate "the user's day" correctly regardless of where they are.
- Profile and settings pages, plus light/dark theme support.

## 🧠 Design Choices

**Why gamification at all?** Habit-formation research consistently points to variable, frequent rewards (quests, badges) and loss-aversion mechanics (streaks) as stronger motivators than a plain checklist. The streak shield specifically exists to soften the all-or-nothing failure mode that makes most streak features feel punishing rather than encouraging.

**TypeScript end-to-end.** Both the React frontend and the Express backend are TypeScript, so the shapes of tasks, quests, and reports stay consistent across the API boundary — fewer "the frontend expected a field that doesn't exist" bugs.

**React + Vite.** Chosen for fast local iteration and a lean production build, with no framework-level opinions getting in the way of a fairly custom, animation-heavy UI (level-up overlays, XP bars, themed Kanban cards).

**Express with a routes / services / jobs split.** Routes stay thin (handle HTTP concerns only); business logic — like badge evaluation or report generation — lives in `services/`; and anything time-based (quest generation, streak checks, report runs) lives in `jobs/` and is triggered by cron rather than by user requests. This keeps user-facing endpoints fast, since heavier computation happens on a schedule, off the request path.

**PostgreSQL, with one migration per feature.** Gamification data is inherently relational (users ↔ tasks ↔ badges ↔ quests ↔ streaks), which suits a relational database well. Each feature shipped with its own migration file rather than one big schema, so features could be added incrementally without rewriting existing tables.

**Redis alongside Postgres.** Used to cache frequently-read, slower-changing data — like leaderboard standings — so high-traffic reads don't hit Postgres on every request.

**Docker Compose for infra.** Postgres and Redis are defined in `docker-compose.yml` so the same database/cache setup runs identically wherever the app is built or deployed.

## 📁 Project Structure

    ├── backend/          # Express API, routes, services, cron jobs, migrations
    │   ├── routes/       # Auth, tasks, quests, badges, leaderboard, reports, calendar
    │   ├── services/     # AI report generation, badge evaluation
    │   ├── jobs/         # Scheduled quests, streaks, and reports
    │   └── migrations/   # Database schema migrations
    └── src/              # React frontend
        ├── components/   # Kanban, XP bar, streak card, Pomodoro, Lo-fi player, etc.
        ├── pages/        # Dashboard, Kanban, Leaderboard, Analytics, Reports, Auth
        └── store/        # Theme state management
